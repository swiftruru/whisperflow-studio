# v1.15.6

Main-screen card consolidation and several queue-flow bug fixes flagged while using the v1.15.5 queue refactor.

## Changed

- **Three cards reduced to two** ([src/renderer/index.html](src/renderer/index.html), [src/renderer/components/queue-panel.js](src/renderer/components/queue-panel.js), [src/renderer/components/controls-bar.js](src/renderer/components/controls-bar.js)) — "Next to Transcribe" was duplicating the current-file info already shown in "Batch Progress" and the queue list. It's now removed; its "Show in folder" button moves into the Batch Progress card's title row as an icon button.

- **Batch Progress only appears while a transcription is actually running** ([src/renderer/components/queue-panel.js](src/renderer/components/queue-panel.js)) — idle / ready / completed states hide the card entirely so the main column stays focused on the queue. Scan summaries are already surfaced in the console log (`scanAppendSummary`, added in v1.15.5).

- **Queue-wide actions moved to the queue card header** ([src/renderer/index.html](src/renderer/index.html), [src/renderer/styles.css](src/renderer/styles.css)) — "Retry Failed" and "Clear Finished" operate on the queue, not the current progress, so they now live next to the queue card's title. The buttons are compacted to 22px tall so they don't dominate the header.

## Fixed

- **"Run" button guard no longer bails on failed-only queues** ([src/renderer/components/controls-bar.js](src/renderer/components/controls-bar.js)) — `getRunnableJob()` in the main process already picks up both `pending` and `failed` jobs, but the renderer's pre-run check only looked at `stats.pending`. That caused a queue of only failed jobs (e.g. after Stop Batch) to show a misleading "nothing to transcribe" toast despite the backend being ready to retry them. The guard now only bails when both pending and failed are zero.

- **Confirm prompt before replaying a deliberately-stopped queue** ([src/renderer/components/controls-bar.js](src/renderer/components/controls-bar.js), [locales/*/dialogs.json](locales/zh-TW/dialogs.json)) — when the only runnable jobs left are ones the user stopped via Stop Batch (detected by `stageMessageKey === 'events:queue.stoppedByUser'`), clicking Run now asks "Restart the stopped items?" instead of silently re-running them. Ignoring the user's earlier stop intent was too aggressive.

- **Scan now re-picks `currentJobId` unconditionally** ([src/main/queue-manager.js](src/main/queue-manager.js)) — if the current job pointer was still pointing at a `failed` / `skipped` item from a previous run, appending a new `pending` scan result did not update the pointer. `getRunnableJob()` then prefers the stale current job (since `failed` is valid there) and Run would transcribe the old failed file instead of the newly scanned one. `scanMedia()` now always calls `setNextCurrentJob()` after merging, so a fresh `pending` always wins over an older `failed`.
