# v1.4.3

Patch release fixing the silent transcription failure caused by missing
`ffmpeg` / `ffprobe` on the packaged app's PATH, and surfacing real
Python exceptions in the UI instead of a generic "轉錄失敗" banner.

## Fixed

- **Packaged macOS / Windows / Linux apps couldn't find `ffmpeg` or
  `ffprobe`.**  GUI apps inherit a minimal PATH from the desktop
  launcher (Finder / Explorer / `.desktop` file), not the user's shell
  PATH.  Tools the user installed via Homebrew, Scoop, or
  `apt --user` were therefore invisible to the Python side, and any
  call to `ffmpeg.probe()` inside `audio/source.py` crashed with
  `[Errno 2] No such file or directory: 'ffprobe'` ~5 seconds into
  every transcription job.

  Phase 3 of v1.4.0 inadvertently dropped the `macPathPrefixes` entry
  from `config.metadata.json :: appRuntime`, which silently disabled
  the PATH-augmentation block in `src/main/main.js`.  This release
  restores it, generalised to all three platforms.

- **The error banner used to hide the real failure reason.**  When
  whisperflow's Python side raised an exception, `bridge/run_cli.py`
  caught it and sent a structured `[WhisperFlowEvent]` error message
  containing the actual text.  But `ipc-handlers.js`'s `run:cli` exit
  handler used a hardcoded "目前檔案的轉錄流程失敗" string and stuffed
  the empty stderr buffer into the details field, hiding everything
  the user needed to know.  Fixed: the banner now shows the structured
  error event's message verbatim.

## Added

- `appRuntime.extraPathPrefixes` block in `python/config/config.metadata.json`
  with sensible defaults per platform:
  - **darwin**: `/opt/homebrew/bin`, `/opt/homebrew/sbin`,
    `/usr/local/bin`, `/usr/local/sbin`
  - **linux**: `/usr/local/bin`, `/usr/local/sbin`,
    `${HOME}/.local/bin`, `/snap/bin`
  - **win32**: `${HOME}/scoop/shims`,
    `${HOME}/AppData/Local/Microsoft/WinGet/Packages`,
    `C:/ProgramData/chocolatey/bin`,
    `C:/Program Files/ffmpeg/bin`
- `validateFfmpeg()` preflight check in `src/main/preflight-checker.js`.
  Looks for `ffmpeg` and `ffprobe` on the augmented PATH and surfaces a
  blocking error at startup if either is missing, with platform-specific
  install instructions:

      • macOS:   brew install ffmpeg
      • Windows: scoop install ffmpeg  /  choco install ffmpeg
      • Linux:   sudo apt install ffmpeg  /  sudo dnf install ffmpeg

- `ERROR_CODES.FFMPEG_NOT_FOUND` for the new check.

## Changed

- `src/main/main.js` — the macOS-only PATH augmentation block is now
  cross-platform.  Reads `appRuntime.extraPathPrefixes[process.platform]`
  from metadata, expands `${HOME}` placeholders, deduplicates against
  the current PATH, and prepends the result to `process.env.PATH`
  before any IPC handler runs.
- `src/main/ipc-handlers.js` — the `run:cli` callback now tracks the
  most recent `[WhisperFlowEvent]` error event in scope and uses its
  `message` field as the user-facing banner text when the Python
  process exits non-zero.  Falls back to `stderrBuffer` or the exit
  code only when there's no structured error available (e.g. segfault).
  The error details field also includes the Python exception class
  name when present (`RuntimeError: …`, `FileNotFoundError: …`).

## Why the v1.4.0 user hit this on a working install

The user had `ffmpeg` and `ffprobe` installed via Homebrew at
`/opt/homebrew/bin/`, and `which ffprobe` worked fine in their
Terminal.  But the packaged `WhisperFlow Studio.app` launched from
Finder didn't see `/opt/homebrew/bin` on its PATH, so the Python
subprocess couldn't find `ffprobe`.  v1.4.3 restores the PATH
augmentation that was meant to handle exactly this case but was
disabled by accident in v1.4.0's Phase 3 cleanup.

## Migration notes

- No user action required.  Reinstall v1.4.3 over the existing
  install (the venv at `~/Library/Application Support/WhisperFlow Studio/.venv`
  and the downloaded models are preserved).
- If you don't have ffmpeg installed system-wide, the new preflight
  check at startup will tell you exactly what to install for your
  platform.
