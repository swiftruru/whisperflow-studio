# v1.15.7

Three layers of file-existence guarding so a deleted media file can no longer derail a batch.

## Fixed

- **Run now validates file existence before handing off to Python** ([src/main/queue-manager.js](src/main/queue-manager.js), [src/main/ipc-handlers.js](src/main/ipc-handlers.js)) вҖ” new `validateRunnableFiles()` runs at the start of `run:cli`. Any `pending` / `failed` job whose file has vanished is marked as `failed` with `stageMessageKey = 'events:queue.fileNotFound'`, and the console logs "з•ҘйҒҺ N еҖӢе·Іж¶ҲеӨұзҡ„жӘ”жЎҲ". Previously the batch would still "start" a ghost job and only crash once Python tried to open the missing path.

- **Mid-batch file deletions are tolerated** ([src/main/queue-manager.js](src/main/queue-manager.js)) вҖ” `startNextJob()` now stat-checks the picked job's file before flipping it to `running`. If the file is gone (deleted between jobs), it auto-marks the job as `fileNotFound`-failed and loops to the next runnable one, bounded by queue length so it always terminates.

- **`getRunnableJob()` skips `fileNotFound` failures** ([src/main/queue-manager.js](src/main/queue-manager.js)) вҖ” these represent terminal "can't run" states rather than retryable ones, so auto-Run no longer loops on them. The per-item Retry button still works if the user restores the file.

- **Scan auto-removes jobs whose files are gone** ([src/main/queue-manager.js](src/main/queue-manager.js)) вҖ” new `removeJobsWithMissingFiles()` runs at the start of `scanMedia()`, right after `dropFinishedJobsQuiet()`. This clears out both prior `fileNotFound` failures and any other silently-deleted files so the queue stays tidy across sessions.

- **i18n** ([locales/*/events.json](locales/zh-TW/events.json)) вҖ” new `events:queue.fileNotFound` ("жӘ”жЎҲдёҚеӯҳеңЁ") shows up in queue item stage messages, and `events:log.skippedMissingFiles` is the console summary when pre-run validation flags any.
