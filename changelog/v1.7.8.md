# v1.7.8

Patch release that polishes the Chinese UI — **translated tab labels and settings section headers** — and stops the install dialog from claiming "finished" when the package manager actually failed silently.

## Fixed

- **Help panel overlapped the titlebar and hid the language toggle on Windows** — the help panel used `position: fixed; inset: 0;` so its backdrop + side sheet covered the entire viewport top-to-bottom. On Windows the 44px titlebar row contains the language toggle, theme toggle, help button, and the native `titleBarOverlay` min/max/close buttons, all of which got visually occluded the moment the panel opened — the user couldn't even tell where the language toggle was any more. The panel is now anchored below the titlebar (`top: var(--titlebar-h)`), so the whole toolbar row — including the native window controls — stays visible and clickable while help is open. macOS and Linux behave identically; the change is purely a fix for overlap, not a layout shift.
- **Sidebar tab labels were untranslated under zh-TW** — `locales/zh-TW/sidebar.json` had `tabs.main`, `tabs.models`, and `tabs.settings` set to the literal English words (`"Main"`, `"Models"`, `"Settings"`) instead of proper Chinese. Only the `tabs.about` entry had been translated (`"關於"`), so under the Chinese UI the four tabs rendered as `Main / Models / Settings / 關於` — a jarring mix. Fixed to `主要 / 模型 / 設定 / 關於`.
- **Settings page section headers surfaced raw config keys** — the Settings tab rendered its two collapsible groups as `[APP_SETTINGS]` and `[SETTING]`, lifted straight from the Python `config.json` section names. Non-technical users have no reason to know what those identifiers mean. The section-rendering code now looks up `settings:sections.<KEY>` in i18n and shows a friendly label instead — `應用程式設定 / Application` for `APP_SETTINGS` and `轉錄參數 / Transcription` for `SETTING`. The raw KEY is still used internally for collapse-state persistence and form submission, just not shown in the heading.
- **Silent package-manager failures are now detected at the install layer** — v1.7.7 added a post-install preflight check that caught the ffmpeg 7z decompress-error case by noticing ffmpeg still wasn't on PATH, but by the time that ran we had already logged `[WhisperFlow] ffmpeg install via scoop finished.` to the console, which made users reasonably assume the install worked and then get confused when the left panel still showed "not found" after pressing **重新檢查**.

  The install runner now accumulates the full stdout+stderr of the package-manager subprocess and scans it for known silent-failure signatures **before** trusting the exit code. When a pattern matches, the promise rejects with a new `PM_SILENT_FAILURE` error code, the IPC handler logs `install failed: <reason>` instead of `finished`, and the dialog's catch path runs normally (red error toast, install buttons re-enabled so the user can pick a different manager).

  Patterns currently covered:
  - **Scoop**: `Failed to extract files from` (7z decompress error), `Please try again or create a new issue` (failure boilerplate), `Hash check failed` (checksum mismatch that some Scoop versions still exit 0 on)
  - **winget**: `Installer failed with exit code` (inner subprocess failure wrapped by an outer winget exit 0), `The operation was cancelled by the user` (UAC prompt cancelled while winget was in `--silent` mode)
  - **Chocolatey**: `The install of <pkg> was NOT successful`

  The list is additive — we can grow it as new silent-failure modes are reported.

## Context for the preflight "not found" confusion

The **重新檢查** button itself was always working correctly: it re-runs the preflight detection, which walks PATH looking for `ffmpeg.exe` / `ffprobe.exe`. When Scoop's 7z extraction fails, Scoop never writes a shim into `~\scoop\shims`, so there's nothing for preflight to find — the "not found" banner was an honest answer. The actual bug was that we told the user the install *finished* when it hadn't, which shifted blame onto the detection code. v1.7.8 says "failed" when the install failed, so the preflight banner and the console log now tell the same story.
