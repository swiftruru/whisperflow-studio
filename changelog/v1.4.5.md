# v1.4.5

Patch release that teaches WhisperFlow Studio about **pyenv-win**
directly and adds a diagnostic log for "找不到 Python 3" failures so
we can fix future detection regressions without guessing.

## Fixed

- **pyenv-win users still hit "找不到 Python 3" on v1.4.4.**  The v1.4.4
  fix (temp-script `.bat` self-report) was correct in theory but
  depended on `where.exe python` actually returning the pyenv-win
  shim — which requires `%USERPROFILE%\.pyenv\pyenv-win\shims` to be
  on the Electron GUI app's inherited PATH.  In practice, the
  packaged app often launches with a snapshot of PATH that's missing
  the shim directory (especially when pyenv-win was installed after
  the user's session started), so `where.exe` returned nothing and
  the resolver fell through to failure.

  v1.4.5 adds two independent recovery paths:

  1. **Direct `.pyenv\pyenv-win\versions` scan** — a new
     `resolveViaPyenvWin` step walks the versions directory,
     picks the highest-numbered install, and returns its
     `python.exe` without depending on `where.exe` or PATH at
     all.  Also honours the `PYENV` / `PYENV_ROOT` environment
     variables for custom pyenv-win locations.
  2. **pyenv-win shim dirs on the augmented PATH** — the
     Windows `extraPathPrefixes` list in
     `python/config/config.metadata.json` now includes
     `${HOME}/.pyenv/pyenv-win/shims` and `${HOME}/.pyenv/pyenv-win/bin`,
     so `where.exe python` sees them even when the OS environment
     didn't.

## Added

- **Python detection trace log.**  `src/main/path-resolver.js` now
  buffers breadcrumbs through each resolver step
  (`resolveSystemPython` → knownPythonPaths → pyenv-win scan →
  self-report → manual PATH walk) and flushes them to
  `<userData>/python-detection.log` on every run.  When a user
  reports detection problems, they can attach this file instead of
  us having to ship a debug build.

  - **macOS**: `~/Library/Application Support/WhisperFlow Studio/python-detection.log`
  - **Windows**: `%APPDATA%\WhisperFlow Studio\python-detection.log`
  - **Linux**: `~/.config/WhisperFlow Studio/python-detection.log`

  The log captures:
  - the `PATH` the main process actually inherited
  - which `knownPythonPaths` entries were checked and whether they
    existed
  - the result of the pyenv-win scan (each `versions/*` directory
    walked)
  - every `where.exe` / `which` result for the shim candidates
  - the stdout / error of every `askPythonForSelfPath` attempt,
    both the direct-exe path and the `cmd.exe /d /s /c` shim path

## Why this regression kept recurring

Each Python install on Windows has its own quirks — python.org
installer, Microsoft Store stub, pyenv-win, conda, scoop, chocolatey,
py-launcher, venv-in-venv — and every fix we've shipped for one of
them has quietly exposed a blind spot in another.  v1.4.5's trace
log is the missing diagnostic layer: next time this breaks, we'll
have the actual failure reason on the first support ticket instead
of having to ask the user to screenshot intermediate state.

## Migration notes

- No user action required.  Reinstall v1.4.5 over the existing
  install; the bundled venv and downloaded models are preserved.
- If preflight still reports "找不到 Python 3", open
  `python-detection.log` from your user data directory (paths
  above) and attach it to your bug report — it will tell us
  exactly which step failed.
