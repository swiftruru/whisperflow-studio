# v1.7.13

Patch release that **widens the "are you sure you want to close while a task is running?" guard** from transcription-only to every long-running operation: model downloads, venv bootstrap, and package-manager installs.

## Fixed

- **Closing the window during a long-running task other than transcription silently aborted it** Рђћ the existing close-guard dialog in `main.js` only fired when the `isRunning` flag was true, and that flag was only set by the transcription run path via the `app:set-running` IPC. Accidentally closing the window while a multi-gigabyte model download, the first-launch venv bootstrap, or a winget / Scoop ffmpeg install was in flight would tear the child process down with no warning and lose all progress. Users reported this was especially bad for model downloads Рђћ clicking an X on the window during a 15-minute `Systran/faster-whisper-large-v2` fetch could mean re-doing the entire download from scratch.

  `main.js` now replaces the boolean `isRunning` with a `busyReasons` `Set` of opaque reason strings (`'transcription'`, `'model-download:large-v2'`, `'venv-init'`, `'pm-install:ffmpeg'`, Рђд). Any IPC handler that starts a long-running operation calls `addBusyReason(...)` before the work and `removeBusyReason(...)` in a `finally` block, so the reason is cleared even on exceptions.

  The window `close` handler no longer checks `isRunning`; it checks `isBusy()` (i.e. `busyReasons.size > 0`). When busy, it picks the highest-priority reason via a small `BUSY_PRIORITY` table (transcription > model download > venv init > pm install), maps it to a localized message via the new `dialogs:closeWhileBusy.reasons.<kind>` i18n path, and shows a confirm dialog that names the specific task in flight instead of always saying "Transcription in progress".

  Reasons that `dialogs:closeWhileBusy.reasons.*` covers:
  - `transcription` Рђћ "УйЅжїёТГБтюежђ▓УАїСИГ / Transcription in progress"
  - `model-download` Рђћ "ТеАтъІСИІУ╝ЅСИГ№╝ѕ{{subject}}№╝Ѕ/ Model download in progress ({{subject}})"
  - `venv-init` Рђћ "Python УЎЏТЊгуњ░тбЃТГБтюет╗║уФІСИГ / Python virtual environment is being created"
  - `pm-install` Рђћ "{{subject}} т«ЅУБЮСИГ / {{subject}} is being installed"
  - `generic` Рђћ fallback for any future reason not yet mapped

  Three IPC handlers are wrapped in this release:
  - `models:download` Рђћ adds `model-download:<name>`
  - `venv:initialize` Рђћ adds `venv-init`
  - `pm:install` Рђћ adds `pm-install:<package>`

  The existing `app:set-running` IPC from the renderer's transcription path is routed through a shim that adds/removes `'transcription'` in the same set, so the old code path keeps working unchanged.

- **Force-close clears every busy reason** Рђћ when the user picks the т╝итѕХжЌюжќЅ / Force close button in the dialog, `busyReasons.clear()` runs before `mainWindow.close()` so re-entering the close handler (which Electron sometimes does when the window has child dialogs) doesn't re-prompt against a now-irrelevant reason.

- **`before-quit` clears every busy reason too** Рђћ quitting via the application menu or Cmd-Q used to leak stale reasons if a task was still registered; they're now cleared proactively so the `close` handler doesn't re-trigger during shutdown.

## Migration notes

No user action required. If you were accidentally losing model downloads by closing the window mid-fetch, v1.7.13 catches you before the teardown and gives you an explicit "keep waiting / force close" choice.
