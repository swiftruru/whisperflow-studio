# v1.7.10

Patch release that fixes the **dark-theme flash on cold boot** that was especially visible on Windows.

## Fixed

- **FOUC: dark theme flashed briefly on every launch before flipping to light** — the CSS defines dark mode as the "no attribute" state and light mode as `[data-theme="light"]`, but `initTheme()` (which reads the saved preference and sets the attribute) runs *after* the renderer's full init sequence: i18n resources, preflight, about-panel, help-panel, update-dialog, model-manager, queue-state, controls-bar, etc. On Windows the init sequence is slower because the Python path scanner walks pyenv / Homebrew / %LOCALAPPDATA% looking for Python 3, so users consistently saw:

  1. Electron's default white/black window for one frame.
  2. The dark CSS palette for several hundred milliseconds while init ran.
  3. A sudden flip to the light cream palette when `initTheme()` finally fired.

  v1.7.10 fixes this in two layers:

  **Renderer side**: a new `src/renderer/theme-boot.js` is loaded as a classic script **in `<head>`**, before `styles.css`, so it runs synchronously before the first paint. It reads `localStorage.theme` (falling back to the OS's `prefers-color-scheme`) and, if the resolved theme is light, writes `data-theme="light"` onto `<html>` right there. By the time the stylesheet is parsed and applied, the correct theme attribute is already in place and the light palette wins the very first paint. The file is plain JS, ~20 lines, no dependencies, and wraps every step in `try/catch` so a broken `localStorage` or missing `matchMedia` can never crash boot.

  **Main side**: `BrowserWindow` now sets `backgroundColor` explicitly based on `nativeTheme.shouldUseDarkColors` (the OS's current dark-mode preference). This matches the cream `--mantle` value for light OSes and the cocoa `--mantle` value for dark OSes, so even the split second of Electron-drawn window background (before any HTML loads) is already in the right colour family for the user's system. A user who overrides the in-app theme to the opposite of their OS preference will still see a brief mismatched flash on the very first paint after that override — acceptable edge case, documented in the source.

- **Windows titleBarOverlay was hardcoded to cream colours** — the native min/max/close buttons on Windows are drawn by Electron's `titleBarOverlay` and their background/symbol colours were hardcoded to `#fbf8ef` / `#1e1a0e` (light theme only). A dark-mode user ended up with a cream title bar stapled to a cocoa-brown app body. Now the overlay colours are picked from `nativeTheme.shouldUseDarkColors` at window-creation time so they match the OS preference (`#28221a` / `#fbf5e6` for dark). Electron doesn't repaint the overlay when the user flips the in-app theme at runtime — the renderer only re-themes the HTML, not the native overlay — so a user who toggles themes after startup still sees the old overlay colour until next launch. Also an acceptable tradeoff, mirrored as a comment next to the switch.
