# v1.7.4

Patch release that fixes a **Windows-only venv bootstrap failure** where the first-launch Python environment setup crashed while upgrading pip.

## Fixed

- **First-launch venv bootstrap failed on Windows with `[VENV_INIT_FAILED] pip.exe exited with code 1`** — the installer was invoking `pip.exe install --upgrade pip` directly, but on Windows this always fails. pip refuses to upgrade itself when driven through `pip.exe` because the `.exe` file is locked by the running process, and prints the well-known guidance:
  > ERROR: To modify pip, please run the following command:
  > `<python> -m pip install --upgrade pip`
  The installer now uses `python -m pip install --upgrade pip` (and, for consistency, `python -m pip install -r requirements.txt`) so the pip upgrade goes through the Python interpreter and doesn't hold a lock on `pip.exe`. This invocation is already how pip itself recommends the self-upgrade on every platform, so macOS and Linux are unaffected but get the exact same code path.

## Migration notes

If you were hit by this on v1.7.3 or earlier, the partial venv left behind at `%APPDATA%\WhisperFlow Studio\.venv` won't be reused — v1.7.4's bootstrap will overwrite it from scratch on next launch. No manual cleanup required.
