# v1.7.7

Patch release that fixes **three compounding bugs** in the Install ffmpeg flow on Windows.

## Fixed

- **winget was reported as "not installed" even when it was** — on Windows 10/11 `winget.exe` lives in `%LOCALAPPDATA%\Microsoft\WindowsApps` as an **App Execution Alias**. These are `APPEXECLINK` reparse points, not real files; `fs.statSync(...).isFile()` returns `false` on them, so the package-manager detection's manual PATH walk silently missed winget every time. The Windows detection path now delegates to `where.exe` first — it's the OS's canonical resolver and correctly follows the alias to the real target — and falls back to the manual walk only if `where` itself is missing. Also added `%LOCALAPPDATA%\Microsoft\WindowsApps` to `appRuntime.extraPathPrefixes.win32` so child processes spawned by WhisperFlow Studio inherit a PATH that includes the alias directory.
- **Close button was dead during an active install** — `closeInstallFfmpegDialog()` had an explicit early-return `if (installing) return;`, so clicking 關閉 / Close did nothing while a package-manager child process was running. Combined with Scoop's auto-bucket-update-on-install behaviour, users reported the dialog locking them in for 15+ minutes with no way to bail out. The guard is gone — the close button now always responds.
- **Closing the dialog didn't re-run preflight** — even after a successful (or failed) install, the left panel's "ffmpeg / ffprobe not found" banner stayed stale until some other preflight trigger fired. `closeInstallFfmpegDialog()` now fire-and-forget-calls `refreshPreflight()` so the banner clears immediately when the install actually worked, and stays visible (correctly) when it didn't.
- **Silent install failures now surface as real failures** — Scoop occasionally exits 0 on a failed 7z extraction (users have reported this with `ffmpeg decompress-error`), and winget's `--silent` mode can report success when the user cancels a UAC prompt. The dialog used to blindly trust the exit code and paint the button with an "Installed" badge, even though nothing ended up on disk. The success path now re-runs preflight after the package-manager command returns and, if the ffmpeg check is still in the error state, throws a synthetic `PM_VERIFY_FAILED` that re-enables the install buttons and shows a bilingual error toast suggesting the user try a different package manager instead.

## Added

- **Cancel install button** — during an active install the 關閉 / Close button relabels itself to 取消安裝 / Cancel install and, on click, kills the running child process through a new `pm:cancel-install` IPC. The runner promise then rejects with a new `PM_INSTALL_CANCELLED` error code, the dialog shows a "cancelled" toast, re-enables the install buttons so the user can pick a different manager, and the Close button returns to its normal behaviour.
- **Cross-process-tree kill on Windows** — a plain `child.kill()` on Windows only takes out the `cmd.exe` shim that `scoop`, `winget`, and `choco` are invoked through, leaving the real install processes running as orphaned grandchildren. The cancel path now uses `taskkill /PID <pid> /T /F` on Windows to walk the process tree and kill every descendant, with `child.kill()` as a best-effort fallback if taskkill fails. macOS and Linux use a plain SIGTERM escalating to SIGKILL after a 2-second grace period.
- **Backdrop click is safer** — clicking the modal backdrop only closes the dialog when idle. During an active install the backdrop click is a no-op, so a stray misclick can't tear down a half-done download; the explicit Cancel install button is the only way to abort.

## Migration notes

No user action required. Windows users who had winget all along and were stuck with Scoop will finally see winget listed as **available** on next launch of v1.7.7. Anyone who got bitten by the Scoop 7z decompress bug in v1.7.6 will now get a clear error instead of a false "Installed" badge.
