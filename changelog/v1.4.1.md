# v1.4.1

Patch release fixing Windows Python detection.

## Fixed

- **Windows: system Python 3.13+ no longer detected as missing.**  The
  preflight panel previously reported "找不到 Python 3" even when
  `python --version` worked fine in PowerShell.  Two root causes:
  - `config.metadata.json :: knownPythonPaths.win32` only listed Python
    3.10 / 3.11 / 3.12, so `Python313\python.exe` wasn't probed.
  - The PATH-iteration fallback only worked when Python's install
    directory was literally on `process.env.PATH` for the GUI app —
    Microsoft Store stubs, py-launcher-only installs, and "didn't tick
    Add to PATH" installs were all invisible.

## Added

- `resolveViaShellWhich()` in `src/main/path-resolver.js`: delegates
  Python discovery to the OS via `where.exe` on Windows and `which` on
  POSIX.  This finds exactly the interpreter the user's shell would,
  matching `python --version` in PowerShell.  Replaces brittle path
  enumeration with a single call to the platform's native lookup.
- Expanded `knownPythonPaths.win32` with:
  - Python 3.13 and 3.14 in the per-user install location
  - Microsoft Store stubs under `WindowsApps/`
  - `Program Files\Python3xx\python.exe` (admin install)
  - `C:\Windows\py.exe` (the universal Python Launcher for Windows)

## Changed

- `resolveSystemPython()` precedence is now: explicit user override →
  `knownPythonPaths` list → shell-which → manual `$PATH` iteration.
  Most installs hit the second or third branch and never need the
  fallback.

## Build

- `package.json` bumped to 1.4.1.
- No Python or renderer code changed; rebuild produces a smaller
  diff vs v1.4.0.

## Migration notes

- No user action required.  Existing installs that were stuck at
  "找不到 Python 3" can either upgrade to v1.4.1, or as a workaround
  on v1.4.0, manually paste the result of `where python` into the
  Settings tab's `pythonPath` field.
