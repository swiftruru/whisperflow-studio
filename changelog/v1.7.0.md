# v1.7.0

Minor release that introduces a full **in-app version update system**, a new **Help panel** with onboarding tour, and a native **application menu** in VS Code style. Plus Windows i18n robustness, a redesigned Settings form layout, and several UI polish fixes.

## Added

### In-app update system

A complete, platform-aware update flow so users no longer have to check the GitHub releases page manually.

- **5-second passive launch check** — on every app start, the updater waits five seconds (so it never blocks the first frame or the Python bootstrap) and then queries `api.github.com/repos/.../releases/latest` anonymously. If the app is already on the latest version the check is completely silent; users only ever see the dialog when something is actually new.
- **Manual "Check for Updates…" entry points** — a new native **application menu** adds a VS Code-style entry (in the app menu on macOS, and in the **Help** menu on Windows/Linux), and a new **Software updates** card on the About tab exposes the same action as an inline button. Manual checks always give feedback: a "checking…" toast, then either "you're on the latest version" or the update dialog.
- **Three-button update dialog** — a themed modal shows the current version, the latest version, a release-notes preview, and three actions:
  - **Update now** — platform-specific, see below.
  - **Skip this version** — persists `updater.skippedVersion` in `settings.json`; future passive checks stay silent for this specific version. Newer versions that come out later re-trigger the dialog automatically.
  - **Remind me later** — closes the dialog without writing any state, so the next passive check will surface the same version again.
- **Platform strategy pattern** — the main-process updater module picks one of two strategies at runtime:
  - **Windows NSIS installer → full auto-update** via `electron-updater`. The dialog switches to a progress bar, downloads the signed installer (SHA-512-verified against `latest.yml`), and on completion offers an **Install & restart** button that calls `autoUpdater.quitAndInstall()`.
  - **macOS DMG / Windows portable / Linux AppImage → redirect** to the GitHub release page via `shell.openExternal`. Because the macOS build is unsigned and portable installers can't overwrite themselves, attempting to auto-install on those platforms would produce broken states; the redirect keeps the experience honest.
- **Modular architecture** — the new [`src/main/updater/`](src/main/updater/) folder splits the concern into single-responsibility files (`config`, `version-compare`, `github-release-checker`, `skip-state`, `platform-strategy`, `strategy-manual-download`, `strategy-electron-updater`, `index`, `updater-ipc`). Adding a future strategy (e.g., Linux AppImage auto-update) is a single new file plus one dispatch line in `platform-strategy.js`.
- **Build pipeline updates** — `electron-builder.yml` now declares a GitHub `publish` provider so electron-builder emits the `latest.yml` / `latest-mac.yml` / `latest-linux.yml` metadata files needed by the auto-updater, while the release workflow continues to use `--publish never` and uploads those metadata files via `gh release upload` so the existing manual release flow is preserved.
- **New `updater` i18n namespace** — 40+ keys covering the menu entry, About-page card, dialog copy, toasts, and progress labels, in both zh-TW and en.

### Help panel & first-launch onboarding tour

A new Help button in the titlebar opens a full in-app help panel with bilingual articles covering first-launch setup, model selection, batch transcription, keyboard shortcuts, and troubleshooting. The panel doubles as the entry point for the first-launch onboarding tour, a five-step guided walkthrough built on a new reusable `spotlight` primitive that highlights UI targets and pins callouts over them. The tour runs automatically once per install (gated by `hasSeenOnboarding` in `settings.json`) and only fires after preflight is unblocked, so first-time users are never walked through a tour while the Python environment is still bootstrapping.

### Application menu

WhisperFlow Studio previously used Electron's default application menu. v1.7.0 installs a custom menu built with `Menu.buildFromTemplate` that preserves all the standard `editMenu` / `viewMenu` / `windowMenu` / `help` roles while adding the **Check for Updates…** entry in the platform-conventional location and a **About WhisperFlow Studio** entry under Help on Windows/Linux that switches the renderer to the About tab via a new `menu:open-about` IPC broadcast. Menu labels intentionally stay in English regardless of UI language, matching VS Code and the platform norms for native OS menus.

## Changed

- **`src/renderer/components/language-toggle.js`** — hardened against missing DOM nodes and early-init races so the toggle no longer throws on cold boot if the renderer finishes parsing before i18n resources land.
- **`src/renderer/lib/i18n.js`** / **`src/renderer/lib/i18n-dom.js`** — defensive `t()` wrapper that detects missing-key echoes (when i18next returns the key itself) and falls back to the element's existing innerHTML, plus BOM stripping in `loadAllResources()`. This was driven by a Windows-only bug where the Settings tab was rendering raw i18n keys (`settings:field.description.X`) instead of the translated strings — traced to a UTF-8 BOM in the locale JSON files after a Windows editor round-trip.
- **Settings form layout** — switched from a cramped `110px / 1fr` two-column grid to a stacked layout where the label, input, and description each get their own full-width row. The old grid squeezed long English field names against the input; the new layout reads comfortably in both locales.
- **Preflight panel button sizing** — the inline action buttons inside preflight rows had `border-radius: 99px` and no `max-width`, which let long English labels (e.g. "Install ffmpeg") push the body copy into a single cramped column on narrow panel widths. Capped the button at `max-width: 120px`, dropped the pill radius to `14px`, and enabled word-wrap so the body text can breathe.
- **`src/main/i18n.js`** — the `NAMESPACES` list now includes `help` and `updater`, bringing the total to 16 feature namespaces with ~547 keys per locale.
- **`settings.example.json`** — new `updater: { skippedVersion: null }` block. Existing installs inherit the default via the settings merge logic in `normalizeLocalSettings`, so no migration is required.
- **About tab layout** — the **Check for updates** action was moved out of the Credits & License card into a dedicated **Software updates** card directly above it. Putting an update action inside a credits block was confusing UX; the new card makes the intent obvious at a glance.
- **`preload/preload.js`** — adds the `window.electronAPI.updater.*` bridge (check, skip, start, install, getStrategy, plus six event listeners) and a `onMenuOpenAbout` hook for the new Help-menu entry.

## Fixed

- **Onboarding tour final step was unclickable** — `finishTour()` was only toggling `aria-hidden` on the tour root, leaving the callout's `.visible` class and inline spotlight styles in place, so the "Finish" button was intercepting clicks against an invisible overlay. Now explicitly clears `.visible`, removes inline styles, and wipes innerHTML.
- **Missing i18n keys on Windows English Settings tab** — raw keys like `settings:field.description.modelName` were leaking into the UI when loaded with UTF-8 BOM; the defensive `t()` wrapper and BOM stripping together make the loader resilient to this.

## Migration notes

No user action required. Reinstall v1.7.0 over the existing install; the bundled venv, downloaded models, and your saved settings (including `uiLanguage`, theme preference, and `hasSeenOnboarding`) are all preserved. The new `updater.skippedVersion` setting defaults to `null`, so the first passive check on v1.7.0 will evaluate the current release against itself and stay silent. Windows NSIS users will start getting real auto-updates from the **next** release (v1.7.1 or later), because v1.7.0 is the first build that ships the `latest.yml` metadata the updater needs.
