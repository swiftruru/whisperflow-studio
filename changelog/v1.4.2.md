# v1.4.2

Patch release fixing Python detection for Windows users running
**pyenv-win** (and any other Python install that exposes itself
through a `.bat` / `.cmd` shim instead of a real `.exe`).

## Fixed

- **Windows pyenv-win users were stuck even after v1.4.1.**  v1.4.1
  added a `where.exe python.exe` fallback, but pyenv-win shims are
  named `python.bat` (not `.exe`) and live in
  `~/.pyenv/pyenv-win/shims/`.  Searching for `python.exe` literally
  would either miss them entirely or land on the Microsoft Store stub
  at `WindowsApps/python.exe`, which fails when invoked headless.
- The Store stub (`WindowsApps/python.exe`) is now correctly identified
  as non-functional and skipped.  Previously it could be returned as
  the "system Python" and then crash venv bootstrap when the user hit
  Initialize.

## Added

- `resolveViaPythonSelfReport()` in `src/main/path-resolver.js`.  For
  every Python-shaped file on PATH (located via
  `where.exe python` / `where.exe python3` / `where.exe py`), the
  resolver now runs:

      <candidate> -c "import sys; print(sys.executable)"

  through the system shell (`cmd.exe` on Windows, plain `sh` on POSIX).
  This is the only approach that works uniformly for *every* Python
  install variant:

  | Install                | Shim type            | Self-report result                                      |
  | ---                    | ---                  | ---                                                     |
  | python.org installer   | `python.exe`         | `…\Programs\Python\Python313\python.exe`                |
  | **pyenv-win**          | `python.bat`         | `…\.pyenv\pyenv-win\versions\3.13.0\python.exe`         |
  | Microsoft Store        | reparse-point stub   | (fails / marketing text) → skipped                      |
  | conda / scoop          | `python.exe` / shim  | the real interpreter inside the env                     |
  | py launcher only       | `py.exe`             | whichever 3.x py picks                                  |

- `listPythonShimCandidates()` does the `where.exe`/`which` enumeration
  with name de-duplication, returning all valid PATH matches instead
  of v1.4.1's "first one wins" behaviour.

## Changed

- `resolveSystemPython()` precedence is now: explicit user override →
  `knownPythonPaths` list → **self-report via shim enumeration** →
  manual `$PATH` walk.
- The shim-search step uses the bare names `python` / `python3` / `py`
  (not `python.exe` / etc), so `where.exe` honours `%PATHEXT%` and
  surfaces `.bat` / `.cmd` / no-extension shims that pyenv-win and
  similar wrappers install.

## Why this exists

Phase 3 of the v1.4.0 rewrite assumed Python on Windows would always
be a `.exe` reachable via PATH.  That assumption breaks for several
mainstream install methods:

- **pyenv-win** (very popular for managing multiple versions): shims
  are `.bat` files
- **python.org installer without "Add to PATH"**: only `py.exe` is
  added to PATH; the real `python.exe` lives in
  `~/AppData/Local/Programs/Python/...`
- **Microsoft Store**: registers `python.exe` as a reparse-point stub
  that opens the Store when run

Asking each candidate `import sys; print(sys.executable)` sidesteps
all of this — Python always knows where its own interpreter lives,
no matter how many layers of shims sit in front of it.

## Migration notes

- No user action required.  Existing v1.4.0 / v1.4.1 installs that
  were stuck at "找不到 Python 3" will work after upgrading to v1.4.2.
- Workaround on older versions: paste the output of
  `python -c "import sys; print(sys.executable)"` into the Settings
  tab's `pythonPath` field manually.
