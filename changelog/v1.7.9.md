# v1.7.9

Patch release that fixes the **winget install → "still not found"** loop on Windows. winget really does install ffmpeg successfully, but until v1.7.9 the Electron main process couldn't see the new PATH entry, so the dialog's post-install verification (added in v1.7.7) would report a false `PM_VERIFY_FAILED` and tell users to try a different package manager.

## Fixed

- **winget install succeeded but post-install verification said "still not found"** — when winget installs a package like `Gyan.FFmpeg`, it drops `ffmpeg.exe` / `ffprobe.exe` as App Execution Aliases into `%LOCALAPPDATA%\Microsoft\WinGet\Links\` and writes that directory to the **User PATH** via the Windows registry. winget even prints "已修改路徑環境變數；重新啟動命令介面以使用新值" to tell you exactly that. The problem: the Electron main process's `process.env.PATH` was captured from its launcher at boot time and doesn't track registry writes. Our post-install preflight check walked the stale PATH, couldn't find ffmpeg, and rejected the install as a silent failure — even though it had actually worked.

  v1.7.9 adds a new `src/main/env-path.js` module with `refreshSystemPathFromRegistry()`. After any successful `pm:install` call — and also at the start of every `app:run-preflight` call as a belt-and-suspenders measure — we query `HKCU\Environment\Path` and `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment\Path` via `reg.exe`, expand any `%USERPROFILE%` / `%LOCALAPPDATA%` references, compose the Machine + User PATH the way Windows itself would, re-prepend our boot-time extraPathPrefixes, and write the result back to `process.env.PATH`. Any subsequent child process (preflight's `where.exe` call, the transcription runner's `ffmpeg` spawn, …) inherits the refreshed PATH and finds the newly installed binary. macOS and Linux are no-op — their env model doesn't have this problem.

- **Preflight's ffmpeg detection couldn't see App Execution Aliases** — `src/main/preflight-checker.js :: findExecutableOnPath` walked PATH with `fs.statSync(...).isFile()`, but App Execution Aliases are `APPEXECLINK` reparse points, not real files, so `isFile()` returns false even when the alias resolves fine. Applied the same `where.exe`-first fix that v1.7.7 did for `package-manager.js :: isExecutableOnPath`, so preflight now correctly follows winget-installed aliases for `ffmpeg` / `ffprobe`.

- **`main.js` PATH augmentation is now in a shared module** — the boot-time prepending of `appRuntime.extraPathPrefixes` used to live inline in `main.js`. Moved to `env-path.js :: applyExtraPathPrefixes` so the new registry-refresh path can re-apply the same prefixes after reading the fresh registry PATH (otherwise a refresh would wipe out our boot-time augmentation).

## Migration notes

No user action required. Windows users who were bitten by the fake `PM_VERIFY_FAILED` error with winget on v1.7.8 — despite winget having actually installed ffmpeg — will now see the install correctly succeed on v1.7.9. If your ffmpeg install on v1.7.8 actually worked (it usually did) you can safely assume it's still there; v1.7.9's preflight will pick it up on first launch without re-running the install.
